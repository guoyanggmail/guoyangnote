## 一、Android 核心

### 1. Handler 机制（字节/腾讯/滴滴/百度 必考）

**Q1: Handler 的整体工作机制？**

> **总结**：Handler 通过 Looper + MessageQueue 实现线程间消息通信，本质是生产者-消费者模型。

**分析**：
- `Handler.sendMessage()` → 消息插入 `MessageQueue`（按时间排序的链表）
- `Looper.loop()` 死循环不断取消息 → `epoll_wait` 阻塞监听，有消息时唤醒
- 取出消息后调用 `msg.target.dispatchMessage()` → 回到 Handler 处理
- 线程隔离由 `ThreadLocal<Looper>` 保证，每个线程最多一个 Looper

---

**Q2: Handler 内存泄漏原因及解决？**

> **总结**：匿名内部类 Handler 持有外部 Activity 引用，未处理完的延迟消息会阻止 Activity 被 GC。

**分析**：
- GC Root 链：MessageQueue → Message → Handler（内部类）→ Activity
- 解决：静态内部类 + `WeakReference<Activity>`；`onDestroy()` 调用 `handler.removeCallbacksAndMessages(null)`

---

**Q3: IdleHandler 是什么，什么时候调用？**

> **总结**：MessageQueue 空闲时（没有待处理消息）才执行的任务，适合做低优先级初始化。

**分析**：
- 触发时机：`MessageQueue.next()` 取不到消息时
- 应用：Glide、LeakCanary 都用它做延迟初始化；启动优化中用于非核心模块延迟加载
- 返回 `true` 保留（继续空闲时触发），`false` 移除（只执行一次）

---

**Q4: 同步屏障（MessageBarrier）原理？**

> **总结**：插入一个 target 为 null 的特殊消息，让队列跳过普通消息优先处理异步消息（如 VSYNC 信号）。

**分析**：
- `ViewRootImpl` 绘制时会 `postSyncBarrier()` 插入屏障，绘制完成后 `removeSyncBarrier()` 移除
- `Choreographer` 的回调消息是异步消息，能越过屏障优先执行
- 保证 UI 绘制优先级最高，避免普通消息延迟渲染

---

### 2. Binder IPC（小米/百度/字节 必考）

**Q5: Binder 为什么只需要一次内存拷贝？**

> **总结**：Binder 通过 mmap 将内核缓冲区映射到接收进程用户空间，发送方一次 copy_from_user 后，接收方直接读取，无需第二次拷贝。

**分析**：
- 传统 IPC（Socket/管道）：用户空间A → 内核缓冲区 → 用户空间B，需 2 次拷贝
- Binder：发送方 copy_from_user 到内核，内核通过 mmap 将同一块物理内存映射到接收方，1 次拷贝

---

**Q6: AIDL in/out/inout/oneway 各代表什么？**

> **总结**：描述参数传输方向；oneway 表示异步非阻塞调用。

| 关键字 | 含义 |
|---|---|
| `in` | Caller→Callee，只拷贝进去 |
| `out` | Callee→Caller，结果写回 |
| `inout` | 双向拷贝 |
| `oneway` | 异步，Caller 不阻塞等待结果 |

---

**Q7: Binder Server 端是多线程还是单线程？**

> **总结**：Server 端维护线程池（默认最多16个），每个请求在独立线程处理，需注意线程安全。

**分析**：
- Binder 线程池默认大小 `BINDER_DEFAULT_THREAD_COUNT = 16`
- 如果 16 个都在忙，第 17 个请求会被挂起等待
- 系统服务（如 AMS）的 Binder 方法都需要加锁保护

---

### 3. View 绘制流程（腾讯/字节/滴滴 必考）

**Q8: View 绘制完整流程？**

> **总结**：DecorView → ViewRootImpl.performTraversals() 依次触发 measure → layout → draw 三大流程。

**分析**：
- `measure`：从根到叶递归测量，`MeasureSpec` = `mode` + `size`（2位模式+30位尺寸）
- `layout`：确定每个 View 的 left/top/right/bottom
- `draw`：drawBackground → draw(canvas) → onDraw → dispatchDraw（子View）→ drawForeground
- VSYNC 信号每 16ms 触发一次，`Choreographer` 接收后驱动 performTraversals

---

**Q9: onCreate/onStart/onResume 里能获取到 View 宽高吗？**

> **总结**：都不能直接获取，View 在 `onResume` 之后才开始真正 measure/layout，需用 `view.post()` 或 `ViewTreeObserver`。

**分析**：
- `view.post(runnable)`：将 Runnable 放入 View 的消息队列，等 View 完成 attach 并执行完第一帧后触发
- 原理：View 未 attach 时 post 到静态 RunQueue，attach 后自动 dispatch，此时宽高已确定

---

**Q10: 屏幕刷新机制，双缓冲/三缓冲是什么？**

> **总结**：双缓冲用前缓冲区显示+后缓冲区绘制避免撕裂；三缓冲解决 Jank，让 CPU/GPU 在 VSYNC 等待期间继续工作。

**分析**：
- 撕裂（Tearing）：显示器读取缓冲区时 CPU 正在写，导致同一帧显示两帧内容
- 双缓冲：VSYNC 到来时交换前后缓冲区 → 解决撕裂，但可能出现 Jank（一帧超时导致等下一个 VSYNC）
- 三缓冲：增加一个缓冲区，CPU/GPU 超时时利用空闲缓冲区继续工作，减少掉帧

---

### 4. 事件分发（滴滴/小米/腾讯/vipkid 必考）

**Q11: 事件分发完整流程？**

> **总结**：Activity → PhoneWindow → DecorView → ViewGroup（dispatchTouchEvent → onInterceptTouchEvent） → View（onTouchEvent），子 View 消费失败则回溯给父 View。

**分析**：
- DOWN 事件：从上到下找消费者，建立 `mFirstTouchTarget` 链
- MOVE/UP 事件：直接发给 `mFirstTouchTarget`，跳过重新分发
- 关键：`onInterceptTouchEvent` 返回 true → 自己消费，不再往下传；子 View 可 `requestDisallowInterceptTouchEvent(true)` 屏蔽父 View 拦截

---

**Q12: 滑动冲突如何解决？**

> **总结**：外部拦截法（推荐）：在父 ViewGroup 的 `onInterceptTouchEvent` 中根据业务判断是否拦截；内部拦截法：子 View 调用 `requestDisallowInterceptTouchEvent`。

**分析**：
- 典型场景：ScrollView 嵌套 RecyclerView、ViewPager 嵌套 ViewPager
- 外部拦截：逻辑在父控件，代码集中，推荐
- 内部拦截：逻辑分散，父控件 DOWN 不能拦截（否则内部拦截失效），不推荐

---

### 5. RecyclerView（字节/滴滴/B站 必考）

**Q13: RecyclerView 四级缓存机制？**

> **总结**：mAttachedScrap（屏幕内） → mCachedViews（滑动缓存，默认2个） → RecycledViewPool（按 ViewType 缓存，默认5个） → 自定义 ViewCacheExtension。

**分析**：

| 级别 | 名称 | 是否复用 onBind | 说明 |
|---|---|---|---|
| 1 | mAttachedScrap | 不需要 | 当前屏幕 View，notifyItemXxx 时用 |
| 2 | mCachedViews | 不需要 | 滑出屏幕的 View，精准复用 |
| 3 | RecycledViewPool | 需要重新 bind | 多个 RV 共用，类型匹配复用 |
| 4 | ViewCacheExtension | 自定义 | 自定义缓存策略 |

**预取（Prefetch）**：`GapWorker` 在主线程空闲时预取即将进入屏幕的 item，提前完成 bind，减少卡顿。

### 6. 性能优化（必考，结合项目经验回答）

**Q14: App 启动优化做过哪些？（结合 Lamour 1.2s 案例）**

> **总结**：冷启动优化三板斧：主线程任务异步化、初始化任务有向无环图调度、延迟非关键初始化。

**分析**：
- **Theme 优化**：SplashActivity 使用 `windowBackground` 消除白屏感知
- **异步启动框架**：将 Application.onCreate 中的初始化按依赖关系构建 DAG，多核并行执行
- **拖延初始化**：非首屏必须的 SDK 用 `IdleHandler` 延迟初始化
- **启动任务分类**：必须主线程的（UI相关）、可异步的（网络/数据库初始化）、可延迟的（统计/推送）
- 实际效果：冷启动从 2.8s → 1.2s（Lamour 项目）

**Q15: 内存泄漏排查和解决？**

> **总结**：LeakCanary 监控 + Android Profiler Heap 分析，核心是找非预期强引用链。

**分析**：
- 常见泄漏场景：
  - 单例持有 Activity Context → 改用 ApplicationContext
  - 静态变量持有 View → 及时置 null
  - Handler 延迟消息 → WeakReference + onDestroy 清空
  - 匿名内部类注册的监听器未取消注册（RxJava、LiveData、EventBus）
  - Bitmap 未 recycle（API 26+ 内存在 native heap，仍需注意）
- 工具：`LeakCanary`（原理：WeakReference + ReferenceQueue，5秒后未 GC 则触发 heap dump）

---

**Q16: Bitmap 内存大小如何计算？**

> **总结**：`宽 × 高 × 每像素字节数 × 屏幕密度比例²`，放错文件夹会被缩放。

**分析**：
- `ARGB_8888`：4字节/像素；`RGB_565`：2字节/像素
- 放 `drawable-xxhdpi`（480dpi）加载到 `xxhdpi`（480dpi）设备：原图尺寸
- 放 `drawable-mdpi`（160dpi）加载到 `xxhdpi`（480dpi）设备：放大 3 倍 → 内存 × 9
- API 26+ Bitmap 像素数据存 Native heap，不计入 Java heap，但仍有 OOM 风险

**Q17: ANR 分析和定位？**

> **总结**：主线程超时：Activity 5s / BroadcastReceiver 10s / Service 20s；通过 `/data/anr/traces.txt` 分析主线程状态。

**分析**：
- 查看 `traces.txt`：找主线程（main）是否在 `BLOCKED / WAITING`，分析锁持有情况
- 常见原因：主线程 IO/网络、死锁、Binder 调用超时（被调 Service 阻塞）
- 花椒项目：通过线程池优化 + 业务异步化，崩溃率从 0.8% → 0.07%

### 7. 组件化 & ARouter（腾讯/滴滴/猿辅导 必考）

**Q18: ARouter 原理？**

> **总结**：编译期用 APT 生成路由表Class，运行时加载路由表，通过路径字符串查找目标 Activity/Fragment/Service 并跳转。

**分析**：
1. 注解处理器在编译期扫描 `@Route`，生成 `ARouter$$Root$$moduleName` 等路由表类
2. `ARouter.init()` 时加载所有路由表到内存 Map
3. `ARouter.getInstance().build("/user/detail").navigation()` → 查 Map → startActivity
4. 支持拦截器（AOP）、参数自动注入（`@Autowired`）、跨模块 Service 调用

### 8. 性能监控（结合 APM 项目经验）

**Q19: 如何搭建 APM 监控体系？（结合花椒/Lamour 项目）**

> **总结**：从采集（Crash/ANR/FPS/内存/网络）→ 上报（批量/压缩/优先级）→ 分析（聚合/告警）三层构建。

**分析**：
- **Crash 采集**：Java Crash 用 `Thread.UncaughtExceptionHandler`；Native Crash 用 `signal` 处理
- **ANR 监控**：主线程打印心跳 + WatchDog 定时检测；或监听 `/data/anr` 文件变化

- 实际项目：集成 Firebase + 自定义日志系统，崩溃率从 0.8% → 0.4%（Lamour）

## 二、Kotlin

### 1. 协程（快手/字节/猿辅导 必考）

**Q20: 协程的本质和工作原理？**

> **总结**：协程是基于状态机实现的"可挂起的函数"，编译器将 suspend 函数转换为带 `Continuation` 回调的状态机，避免回调嵌套。

**分析**：
- `suspend` 函数编译后新增 `Continuation` 参数（CPS变换）
- 挂起点（`await/delay`）保存当前状态到 `Continuation`，将控制权交回调用方
- 恢复时调用 `continuation.resume(result)` 从挂起点继续执行
- 协程是**用户态**的，调度不经过内核，切换成本远低于线程

---

**Q21: launch 和 async 的区别？**

> **总结**：`launch` 返回 Job，fire-and-forget；`async` 返回 `Deferred<T>`，需 `.await()` 获取结果，用于并发任务汇聚。

```kotlin
// 串行
val result1 = api.getData1() // 等待
val result2 = api.getData2() // 等待

// 并行 (async)
coroutineScope {
    val d1 = async { api.getData1() }
    val d2 = async { api.getData2() }
    val result = d1.await() + d2.await() // 同时等待两个
}
```

---

**Q22: Flow 冷流和热流的区别？StateFlow vs SharedFlow？**

> **总结**：Flow 是冷流（每次 collect 重新执行）；StateFlow/SharedFlow 是热流（独立于 collector 存在）。

| | Flow | StateFlow | SharedFlow |
|---|---|---|---|
| 类型 | 冷流 | 热流 | 热流 |
| 初始值 | 无 | 必须有 | 无 |
| 重放 | 每次重新执行 | 最新值 1 个 | 可配置 replay |
| 场景 | 一次性数据流 | UI 状态 | 事件总线 |

---

**Q23: 协程异常处理？SupervisorJob 和普通 Job 的区别？**

> **总结**：普通 Job 的子协程异常会取消同级所有协程；SupervisorJob 子协程异常互不影响。

```kotlin
// SupervisorJob：一个子协程失败不影响其他子协程
val supervisor = SupervisorJob()
val scope = CoroutineScope(Dispatchers.IO + supervisor)

scope.launch { /* 失败也不影响下面 */ }
scope.launch { /* 继续正常运行 */ }

// CoroutineExceptionHandler 只对 launch 有效，不对 async 生效
val handler = CoroutineExceptionHandler { _, e -> log(e) }
```

---

**Q24: withContext 和 flowOn 的区别？**

> **总结**：`withContext` 切换当前协程执行上下文；`flowOn` 切换 Flow 上游（生产者）的执行上下文，不影响下游（消费者）。

```kotlin
flow {
    emit(fetchData()) // 在 IO 线程执行
}
.flowOn(Dispatchers.IO) // 仅影响上游 emit
.collect { // 在 Main 线程收集
    updateUI(it)
}
```

### 2. Kotlin 语言特性

**Q25: inline/noinline/crossinline 的区别？**

> **总结**：`inline` 内联函数体（消除 Lambda 对象开销）；`noinline` 标记不内联的 Lambda；`crossinline` 允许 Lambda 被传递给非内联上下文（禁止 `return`）。

**分析**：
- `inline` 适合高阶函数，避免每次调用创建 Function 对象
- Lambda 中 `return` 直接从外部函数返回（非局部返回），`crossinline` 禁止这个行为

---

**Q26: Kotlin 协变（out）和逆变（in）？**

> **总结**：`out T`（协变）只能读不能写，子类型关系保持；`in T`（逆变）只能写不能读，子类型关系反转。

```kotlin
// out：Producer<Cat> 可以当 Producer<Animal> 用
interface Producer<out T> { fun produce(): T }

// in：Consumer<Animal> 可以当 Consumer<Cat> 用
interface Consumer<in T> { fun consume(t: T) }
```

---

**Q27: Kotlin?（可空类型）的原理？**

> **总结**：编译期检查 + 字节码层面通过 `@Nullable/@NonNull` 注解 + NPE 提前检查实现，运行时本质还是 Java 对象。

**分析**：
- `?.`（安全调用）编译为 null 检查 + 条件调用
- `!!`（非空断言）编译为 null 检查 + 抛 `NullPointerException`
- `?:` （Elvis）编译为三元表达式

---

### 3. Kotlin 与 Java 互操作

**Q28: Kotlin 调用 Java 常见问题？**

> **总结**：主要是平台类型（Platform Type）导致的空安全问题，Kotlin 无法确定 Java 方法是否返回 null。

**分析**：
- Java 方法返回值在 Kotlin 中变成 `T!`（平台类型），编译器不检查 null
- 解决：给 Java 代码加 `@Nullable/@NonNull` 注解
- Kotlin 的 companion object 方法如果 Java 调用需要加 `@JvmStatic`；属性需要 `@JvmField` 去掉 getter/setter

## 三、Java

### 1. 多线程与锁（快手/字节/滴滴 必考）

**Q29: synchronized 底层原理？**

> **总结**：对象头 Mark Word 记录锁状态，JVM 通过锁升级（偏向锁 → 轻量级锁 → 重量级锁）优化性能。

**分析**：
- 偏向锁：第一次获取，存线程ID到 Mark Word，下次无CAS
- 轻量级锁：CAS 自旋获取，适合竞争少的场景
- 重量级锁：OS Mutex，线程挂起，适合竞争激烈
- `synchronized` 方法：`ACC_SYNCHRONIZED` 标志；`synchronized` 块：`monitorenter/monitorexit` 指令

---

**Q30: volatile 的作用？能保证原子性吗？**

> **总结**：保证可见性（内存屏障）和有序性（禁止指令重排），但不保证原子性（`i++` 仍然线程不安全）。

**分析**：
- 可见性：写 volatile 变量时强制刷新主内存；读时强制从主内存读
- 内存屏障：写前插 StoreStore 屏障，写后插 StoreLoad 屏障
- 典型用法：单例双重检查锁（DCL）中修饰 `instance`，防止指令重排导致拿到未初始化对象

---

**Q31: HashMap 原理及扩容机制？**

> **总结**：数组 + 链表/红黑树，负载因子 0.75，超过 `capacity × 0.75` 时 2 倍扩容，链表长度超 8 且数组长度 ≥ 64 时树化。

**分析**：
- hash 扰动：`(h = key.hashCode()) ^ (h >>> 16)`，让高位也参与散列，减少碰撞
- 扩容：创建 2 倍大数组，rehash 所有元素（1.8优化：只需判断高位bit是否为1，决定是否移到 `原位 + oldCap`）
- 线程不安全：1.7 头插法扩容会死循环；1.8 尾插法无死循环，但仍有数据覆盖问题 → 用 `ConcurrentHashMap`

---

**Q32: ConcurrentHashMap 1.8 实现原理？**

> **总结**：取消分段锁，用 CAS + synchronized 锁单个桶头节点，分散锁粒度；扩容时并发迁移，其他线程帮助扩容。

**分析**：
- put：CAS 插入空桶；非空则 synchronized 锁链表头节点
- 扩容：ForwardingNode 标记迁移完成的桶，其他线程 put 时检测到会一起协助扩容
- size()：`baseCount` + `CounterCell[]` 分散计数，避免全局锁

---

**Q33: JVM 垃圾回收算法？**

> **总结**：新生代用复制算法（两个 Survivor），老年代用标记-整理；GC Root 可达性分析判断对象是否存活。

**分析**：
- GC Root：栈上引用的对象、静态变量、JNI 引用
- 新生代：Eden:S0:S1 = 8:1:1，Minor GC 后存活对象复制到 Survivor，年龄+1
- 年龄达到阈值（默认15）→ 晋升老年代 → 老年代满 → Full GC（Major GC）
- Android 的 ART：分代GC + 并发GC（CC GC，减少 STW 时间）

---

**Q34: 双亲委派机制及 Android 的类加载？**

> **总结**：类加载先委托父 ClassLoader，找不到再自己加载，防止核心类被替换；Android 用 `PathClassLoader/DexClassLoader` 加载 dex 文件。

**分析**：
- Android 类加载链：BootClassLoader → PathClassLoader（App）
- `PathClassLoader`：只能加载已安装 App 的 dex
- `DexClassLoader`：可以加载任意路径的 dex，热修复/插件化用
- 热修复原理：自定义 ClassLoader，将修复 dex 插到 `dexElements[]` 数组最前面

### 2. 集合与数据结构

**Q35: ArrayList vs LinkedList 的区别？**

> **总结**：ArrayList 随机访问 O(1) 但插入删除 O(n)；LinkedList 插入删除 O(1) 但随机访问 O(n)，内存占用更高。

**分析**：
- Android 推荐用 `SparseArray`（代替 `HashMap<Int,Object>`）和 `ArrayMap`（代替 `HashMap<K,V>`）
- `SparseArray`：int数组 + Object数组，二分查找，无装箱，内存优化
- `ArrayMap`：hash数组 + kv数组，适合元素少的场景

## 四、React Native

**Q36: React Native Bridge 架构原理？新旧架构对比？**

> **总结**：旧架构 JS/Native 通过 Bridge 异步 JSON 通信，是性能瓶颈；新架构用 JSI（C++ 直接内存访问）替代 Bridge，实现同步调用。

**分析**：
- **旧架构**：JS线程 ←→ Bridge（JSON序列化）←→ Native线程，异步通信，大量数据传输时卡顿
- **新架构**：
  - **JSI**：JS 直接持有 C++ 对象引用，无需序列化，可同步调用
  - **TurboModules**：按需加载原生模块（旧架构启动时全加载）
  - **Fabric**：新渲染引擎，UI 计算可在多线程进行
  - **Hermes**：Facebook 优化的 JS 引擎，启动速度 ↑40%，内存 ↓18%

**Q37: RN 性能优化做过哪些？**

> **总结**：减少 JS/Native Bridge 通信量；列表用 `FlatList`；避免不必要渲染（`React.memo`/`useMemo`/`useCallback`）。

**Q38: RN 热更新（CodePush）原理？**

> **总结**：打包时将 JS Bundle 和资源文件上传到 CodePush 服务器，App 启动时检测版本差异，增量下载 diff 包并在本地合并替换，下次启动生效。

**分析**：
- 只能更新 JS 层，Native 代码改动必须发版
- 增量更新：只下载改变的文件（diff），减少下载体积
- 安全：包含 hash 校验，防止篡改

**Q40: 直播秒开优化方案？（Lamour/CamChat 项目，首帧 <300ms）**

**Q41: 如何从 0 到 1 设计一个 IM 消息系统？（Lamour 消息到达率 99.9%）**

> **总结**：消息可靠性靠 ACK + 重传机制；顺序性靠序列号；离线消息靠服务端存储 + 上线拉取。

**分析**：
- **协议**：WebSocket（全双工，低延迟）
- **可靠性**：客户端发送带 msgId → 服务端返回 ACK → 超时未收 ACK 则重发（最多3次）
- **去重**：服务端对 msgId 幂等处理
- **顺序**：每个会话维护 seq 序列号，客户端按 seq 排序展示
- **离线消息**：上线时拉取离线 seq 范围内的消息
- **心跳**：30s 发送 Ping，超时无 Pong 则重连

**Q44: 手写快速排序**
```kotlin
fun quickSort(nums: IntArray, left: Int, right: Int) {
    if (left >= right) return
    val pivot = partition(nums, left, right)
    quickSort(nums, left, pivot - 1)
    quickSort(nums, pivot + 1, right)
}

fun partition(nums: IntArray, left: Int, right: Int): Int {
    val pivot = nums[right]
    var i = left
    for (j in left until right) {
        if (nums[j] <= pivot) {
            nums[i] = nums[j].also { nums[j] = nums[i] }
            i++
        }
    }
    nums[i] = nums[right].also { nums[right] = nums[i] }
    return i
}
```
> 平均 O(nlogn)，最坏 O(n²)（已排序时），空间 O(logn) 递归栈

        prev = curr

**Q46: K 个一组反转链表（猿辅导）**
```kotlin
fun reverseKGroup(head: ListNode?, k: Int): ListNode? {
    var node = head
    var count = 0
    while (node != null && count < k) { node = node.next; count++ }
    if (count < k) return head // 不足 k 个，不翻转

    var prev: ListNode? = null
    var curr = head
    repeat(k) {
        val next = curr!!.next
        curr.next = prev
        prev = curr
        curr = next
    }
    head!!.next = reverseKGroup(curr, k) // head 翻转后变尾节点
    return prev
}
```

**Q47: 二叉树层序遍历（按层打印，快手必考）**
```kotlin
fun levelOrder(root: TreeNode?): List<List<Int>> {
    val result = mutableListOf<List<Int>>()
    if (root == null) return result
    val queue: Queue<TreeNode> = LinkedList()
    queue.offer(root)
    while (queue.isNotEmpty()) {
        val level = mutableListOf<Int>()
        val size = queue.size  // 关键：记录当前层节点数
        repeat(size) {
            val node = queue.poll()!!
            level.add(node.`val`)
            node.left?.let { queue.offer(it) }
            node.right?.let { queue.offer(it) }
        }
        result.add(level)
    }
    return result
}
```

**Q48: 最长回文子串（小米必考）**
```kotlin
fun longestPalindrome(s: String): String {
    var start = 0; var maxLen = 1
    fun expand(l: Int, r: Int) {
        var left = l; var right = r
        while (left >= 0 && right < s.length && s[left] == s[right]) {
            if (right - left + 1 > maxLen) {
                start = left; maxLen = right - left + 1
            }
            left--; right++
        }
    }
    for (i in s.indices) {
        expand(i, i)     // 奇数长度
        expand(i, i + 1) // 偶数长度
    }
    return s.substring(start, start + maxLen)
}
```

**Q52: 生产者消费者模型（字节必考）**
```kotlin
class ProducerConsumer {
    private val queue = LinkedList<Int>()
    private val maxSize = 10
    private val lock = ReentrantLock()
    private val notFull = lock.newCondition()
    private val notEmpty = lock.newCondition()

    fun produce(item: Int) {
        lock.lock()
        try {
            while (queue.size >= maxSize) notFull.await()
            queue.offer(item)
            notEmpty.signal()
        } finally { lock.unlock() }
    }

    fun consume(): Int {
        lock.lock()
        try {
            while (queue.isEmpty()) notEmpty.await()
            val item = queue.poll()
            notFull.signal()
            return item
        } finally { lock.unlock() }
    }
}
```
