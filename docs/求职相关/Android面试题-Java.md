# Android面试题-Java基础&多线程&集合篇

> 本文总结了 Android 面试中高频考察的 Java 基础理论（泛型、类加载、引用、异常）、多线程与并发（锁机制、线程池、内存模型）、以及常用数据结构与集合（HashMap、ConcurrentHashMap 等），通过表格对比和精简的要点帮助您快速掌握核心知识。

## Java 基础

### 泛型与类型擦除

**泛型**是在定义类、接口和方法时附带类型参数，其优点包括：**更安全**（编译时强类型检查）、**更简洁**（消除强转）、**更通用**（适用于多种类型）。

**类型擦除**：
泛型本质上是 `Java` 编译器的语法糖。为了向下兼容，`JVM` 和 `Class` 文件不提供泛型支持，编译器会擦除代码中的泛型信息（保留在类常量池的属性中）。

**Kotlin 中对泛型的补充**：
- `inline`（内联函数）：编译器将函数字节码插入调用处。
- `reified`（实化类型参数）：配合 `inline`，在插入的字节码中用确切类型代替类型实参，解决类型擦除问题。

### 协变与逆变

| 关键字 | 类型 | 概念与作用 | 限制 |
|---|---|---|---|
| `? extends` / `out` | **协变** | 声明集合上限 | 修饰的泛型集合**能读不能写** |
| `? super` / `in` | **逆变** | 声明集合下限 | 修饰的泛型集合**能写不能读** |

### Java 对象生命周期

对象的生命周期分为 7 个阶段：
1. **创建 (Created)**：分配内存，构造对象。
2. **应用 (In Use)**：至少有一个强引用指向它。
3. **不可见 (Invisible)**：有强引用但在程序中无法访问（如方法栈帧弹出后的局部变量）。
4. **不可达 (Unreachable)**：没有任何强引用指向。
5. **收集 (Collected)**：准备 GC 前执行 `finalize()`。
6. **终结 (Finalized)**：等待 GC。
7. **释放 (Deallocated)**：GC 回收完成。

### 类加载机制

类加载过程包含以下阶段：

| 阶段 | 描述 |
|---|---|
| **加载** | 读入 `.class` 二进制流到内存，实例化 `Class` 对象。懒加载机制，只在首次使用时加载。 |
| **链接 - 验证** | 二进制流的合法性和安全性校验。 |
| **链接 - 准备** | 为类静态变量在方法区分配内存并赋默认初始值。 |
| **链接 - 解析** | 将符号引用（如类名、方法名）替换为直接引用（内存地址）。 |
| **初始化** | 执行类构造器对静态变量和静态代码块进行赋值。 |

**常见类加载器**：
- `PathClassLoader`：只能加载当前应用包内的 `dex` 文件。
- `DexClassLoader`：可以加载任意位置的 `dex` 文件（如插件化/热修复）。
- `BaseDexClassLoader`：核心逻辑实现，持有 `DexPathList`，加载时遍历 `element` 数组匹配目标类进行加载。

**双亲委派机制**：
加载类时先将请求逐级向上委托到 `BootStrapClassloader`，若父加载器找不到再逐级向下尝试加载。**优势：避免类的重复加载，并保护系统核心类库安全（防止被篡改）**。

### 四种引用类型

| 引用类型 | 特点 | GC 回收时机 | 适用场景 |
|---|---|---|---|
| **强引用 (Strong)** | 最常见，如 `Object obj = new Object()`。 | 内存不足抛 OOM 也不回收。需显式置 `null` 才能被回收。 | 核心对象维持生命周期 |
| **软引用 (Soft)** | 内存敏感的高速缓存。 | 当内存空间不足时，进行垃圾回收时回收。 | 本地图片缓存、对象池 |
| **弱引用 (Weak)** | 拥有比软引用更短暂的生命周期。 | 只要发生 GC，不管内存是否充足，都会强制回收。 | 避免 Handler 或 ThreadLocal 内存泄漏 |
| **虚引用 (Phantom)** | 无法通过虚引用获取对象本身。必须与 `ReferenceQueue` 联合使用。 | 任何时候都可能被回收。 | 跟踪对象的 GC 活动，清理底层资源 |

### 内存泄漏与排查

**本质**：本该消亡的短生命周期对象（如 `Activity`）被仍在活跃的长生命周期对象强引用导致无法回收。

| 泄漏场景 | 原因分析 | 解决方案 |
|---|---|---|
| **静态变量 / 单例** | 静态成员生命周期同 App，若持有 Activity 会引发生命周期倒挂。 | 置空静态引用；传参改用 `Application Context`；使用弱引用。 |
| **内部类 / 匿名内部类**| 如 `Handler` 或 `Thread` 隐式持有外部类实例。 | 改用静态内部类 + 弱引用持有外部类，页面销毁时移除消息/停止线程。 |
| **集合类未清理** | 全局集合不断添加对象，未使用后未清理。 | 页面退出时及时 `remove()` 并置空集合。 |
| **广播 / 回调未解绑** | 注册了观察者或 EventBus，但销毁时未反注册。 | 在 `onDestroy` 中成对解绑注册。 |

### JVM 内存模型与 GC 机制

Dalvik/ART 虚拟机的内存划分：
- **线程私有**：虚拟机栈（保存局部变量和方法调用栈帧）、本地方法栈（Native 调用）、程序计数器（记录指令行号）。
- **线程共享**：堆区（实例对象存放地，GC 主要区域）、方法区（类信息、常量池、静态变量）。

**GC 垃圾回收算法**：

| 算法 | 原理 | 优缺点 | 适用代区域 |
|---|---|---|---|
| **复制清除** | 将存活对象平移复制到另一块区域，随后清空当前区块。 | 无碎片；但空间利用率仅一半。 | **新生代**（存活对象少，垃圾多） |
| **标记清除** | 遍历堆扫描可达对象标记，随后清除未标记对象。 | 原地清理不浪费空间；容易产生内存碎片。 | **老年代**（存活对象多，变动较少） |
| **分代收集** | 结合对象寿命特征，新生代使用复制清除，老年代使用标记清除/整理。 | 系统化提高整体 GC 效率。 | JVM 整体采用策略 |

### equals() 与 hashCode()

- `==` 判断基本数据类型的值，或引用数据类型的内存地址。
- `equals()` 用于业务逻辑上判断两个对象是否“相等”。
- `hashCode()` 用于提升散列表（如 HashMap）的查询效率（O(1) 定位桶位）。
- **规则约定**：`equals()` 相等的两个对象，其 `hashCode()` 必须相等；`hashCode()` 相等的对象，`equals()` 不一定相等（哈希冲突）。

### 异常处理 (Exception & Error)

两者均继承自 `Throwable`。
- **Error**：虚拟机层面的严重错误（如 `OutOfMemoryError`、`StackOverflowError`），程序无法处理。
- **Exception**：程序层面的异常。
  - **Checked Exception（编译时异常）**：必须使用 `try-catch` 捕获或显式 `throws` 抛出（如 `IOException`）。
  - **Unchecked Exception（运行时异常）**：逻辑错误导致的异常（如 `NullPointerException`、`IndexOutOfBoundsException`），通常由于编码不善导致。

## 多线程与并发

### 线程与进程

- **进程**是资源分配和系统调度的独立单位，通常指一个正在运行的应用。
- **线程**是 CPU 调度的最小单位，属于进程内部的执行流。一个进程至少包含一个线程（主线程）。

### 阻塞线程的控制方法

| 方法 | 所属类 | 描述 | 是否释放锁 |
|---|---|---|---|
| `sleep()` | `Thread` | 让线程进入阻塞挂起状态，倒计时结束苏醒。 | **不释放锁** |
| `wait()` | `Object` | 让线程进入等待队列，需与其他线程通信（`notify`）。 | **释放锁** |
| `join()` | `Thread` | 在 A 线程中调用 B.join()，A 会阻塞直到 B 执行完毕。 | 不释放锁 |
| `yield()` | `Thread` | 主动让出 CPU 执行权，使线程回到就绪态排队。 | **不释放锁** |

### 线程安全与锁机制

**线程安全三要素：**
1. **原子性**：操作不可被打断，要么全成功要么全失败。
2. **可见性**：一个线程对共享变量修改，其他线程立即可见。
3. **有序性**：防止编译器或 CPU 进行指令重排。

#### synchronized 与 volatile 比较

| 特性 / 关键字 | synchronized | volatile |
|---|---|---|
| **作用级别** | 变量、方法、代码块 | 仅修饰变量 |
| **原子性** | **保证**原子性 | 不保证原子性（如 `i++` 非原子操作） |
| **可见性与有序性**| 保证可见性、有序性 | 保证可见性（强制刷新主存），防重排序（内存屏障） |
| **阻塞与性能** | 重量级，可能引起线程阻塞切换 | 轻量级，不会阻塞线程 |

#### JDK 1.8 中 synchronized 的锁升级优化：
**无锁 -> 偏向锁 -> 轻量级锁 (自旋锁) -> 重量级锁**，锁膨胀为单向不可逆以保证性能。

### 死锁 (Deadlock)

**特征与条件**：
1. 互斥使用（资源独占）。
2. 不可抢占（资源只能主动释放）。
3. 占有且等待（拿着一个资源不放，还去请求另一个）。
4. 循环等待（形成 A 等 B，B 等 C，C 等 A 的闭环）。

**解决方案**：规定统一的加锁顺序；使用带有超时的 `tryLock()` 放弃机制。

### 线程池 (ThreadPoolExecutor)

| 种类 | 特点与工作机制 | 适用场景 |
|---|---|---|
| **newFixedThreadPool** | 仅包含固定数量的核心线程，队列无界。 | 控制并限制系统最大并发量的场景。 |
| **newCachedThreadPool** | 无核心线程，最大线程无穷大，空闲 60s 回收。可灵活扩容。 | 大量耗时短且突发的异步任务。 |
| **newSingleThreadExecutor**| 只有 1 个核心线程，所有任务排队串行执行。 | 需保证多任务顺序执行，无需同步的场景。 |
| **newScheduledThreadPool** | 核心线程固定，最大线程无界，支持延时或周期性执行。| 定时任务或延迟任务调度。 |

> **注意**：`Executors` 提供的默认工厂方法容易因为无限队列堆积或无限创建线程导致 `OOM`。生产环境推荐手动使用 `ThreadPoolExecutor` 构造，并明确指定核心数/最大线程数和阻塞队列的最大长度。

## 数据结构与集合

### HashMap 原理

- **底层结构**：数组 + 单链表（在 Java 1.8 中，当链表长度 > 8 且数组容量 >= 64 时转化为**红黑树**，提升极端冲突下的查询效率至 O(log N)）。
- **容量与扩容**：初始容量必须是 `2的幂次方`（为了利用位与运算 `hash & (n-1)` 高效替代取模运算定位下标）。默认负载因子为 `0.75`。
- **线程安全性**：`HashMap` 是非线程安全的，并发操作（尤其是 JDK 1.7 的头插法扩容）会导致链表成环死循环，引发 CPU 100%。多线程环境下请使用 `ConcurrentHashMap`。

### Android 特有集合：SparseArray 与 ArrayMap

为优化移动端内存而设计，通常用来替代较小规模的 `HashMap`，因为不需要包装 `Node/Entry`。

| 集合类型 | 底层数据结构 | 优势与劣势 | 适用场景 |
|---|---|---|---|
| **SparseArray** | 双数组：一个存 `int` key，一个存 `Object` value。 | 避免自动装箱，内存占用极小。查找使用二分查找法，查询速度 O(log N)。 | Key 为整型、数据量几百以内的场景。 |
| **ArrayMap** | 双数组：一个存 Hash 值，一个相邻存放 Key、Value。 | 比 HashMap 节省大量内存。查询亦为二分查找。 | Key 需要是对象形态、数据量较小的场景。 |

### ConcurrentHashMap 并发优化

- **JDK 1.7 方案**：**分段锁 (`Segment` 数组)**。将 HashMap 拆成多段，每段加一份 `ReentrantLock`。并发度等于 Segment 数量。
- **JDK 1.8 方案**：抛弃分段锁，改用 **`CAS` + `synchronized` Node 节点（即桶位首节点）锁**。锁的粒度细化到了每一个桶位，大幅提升极高并发量下的吞吐性能。内部抛弃分段结构，恢复到与 HashMap 一致的 数组+链表/红黑树 的结构。

### CopyOnWriteArrayList

- **原理**：写时复制。写操作（增删改）时加锁，拷贝出一份新数组进行修改，修改完再将集合内部的数组引用指向新数组。读操作完全无锁。
- **特点**：支持读多写少、不允许使用迭代器修改内部元素的集合场景。完全避免了 `ConcurrentModificationException`，但可能会读到刚写入之前的旧数据。

### 其他集合特性总结

| 集合实现 | 特点 |
|---|---|
| **LinkedHashMap** | 继承 HashMap，用双向递归链表穿起来所有的 Node。支持按插入顺序或**访问顺序**（将最新访问的移至队尾，做 LRU 缓存的底层数据结构核心算法）遍历。 |
| **WeakHashMap** | 其 Entry 对象继承自 `WeakReference`。触发局部 GC 机制且引用没被强引用时，没人用的老条目会被 JVM 和内部机制悄悄移除。合适做系统级弱缓存。 |
