# 什么是 Agent 兵团？— 飞天闪客 Agent 集群进化逻辑

*基于B站UP主「飞天闪客」视频《什么是 Agent 兵团？一口气搞懂 Agent 集群的进化逻辑！》整理*

---

## 核心观点

> **从单个 Agent 到 Agent Team（兵团），本质上是在解决"上下文灾难"——一个 Agent 装不下，就来多个并行，各干各的，最后汇总。**

---

## 进化路径：三级跳

| 阶段 | 发生了什么 | 问题在哪 |
|------|-----------|---------|
| **LLM** | 大模型参数不断增加，达到了博士级水平 | 只能说话，不能行动，像废物 |
| **Agent** | 给大模型装上中间代理程序，可以调用工具 | 单个 Agent 遇到复杂任务，速度慢，某一环崩了就全崩 |
| **Agent Team** | 多个 Agent 并行工作，各干各的，最后汇总 | 需要工程化协调，但效率大幅提升 |

---

## 第一层：最简单的多 Agent——开多个窗口

在 Claude Code 里多开几个窗口，每个窗口就是一个 Agent。

> 演示：让它用扎娃写一个 Hello World → 再开一个窗口执行另一个任务 → 这就是最原始的多 Agent 场景。

**但这是"假多 Agent"**——你手动开的，各自独立，不共享上下文，不像真正的协作。

---

## 第二层：SubAgent——让 Agent 自己开 Agent

把"开多个窗口"这件事变成**由主 Agent 自动完成**：

- 主 Agent 接到大任务 → 自己拆解 → 分配给多个 SubAgent 并行执行
- SubAgent 之间**不共享上下文**，也**不互相通信**
- 更像一个个临时工，干完就走

---

## 第三层：Agent Team——真正协作的 Agent 集群

SubAgent 像是临时工，Team 才是真正的协作：

| 特点 | SubAgent | Agent Team |
|------|---------|------------|
| 上下文 | 各自独立，不共享 | 主 Agent 统一协调 |
| 互相通信 | 不通信 | 可以共享信息 |
| 失败处理 | 各自崩各自 | 一个崩了，其他继续，主 Agent 调度修复 |
| 任务分配 | 接到就干 | 主 Agent 规划并行策略 |

---

## 实战演示：论文深入研究

复杂任务：读一篇论文 → 找出所有参考文献 → 逐篇读一遍 → 总结摘要和与原论文的关系 → 生成完整报告

**单个 Agent 串行执行**：慢，中途 Token 达到上限就卡死。

**Keyme Agent 集群执行**（国内产品，比 OpenClaw 更早实现这个思路）：
- 主 Agent 先做整体调度
- 分出 **9 个并行的 SubAgent** 同时工作
- 像老板看员工一样，**实时看到每个 Agent 的进度、当前步骤、甚至思考过程**
- 每个 Agent 有自己的名字、头像、工作介绍
- 其中一个失败 → 自己尝试修复 → 不影响其他 Agent
- 最终输出：**80 多篇参考文献的 HTML 报告**

---

## Agent Team 的"暗黑老板思维"

| 老板理论 | Agent 对应 |
|---------|-----------|
| 任何一个员工都可以被替代 | 任何一个 Agent 失败不影响整体 |
| 只想看进度和结果 | 主 Agent 汇总报告，不关心中间细节 |
| 细节透明可查 | 可以展开每个 Agent 的思考过程 |

> **"怪不得老板们还有个不当人的理论：通过一些手段，让组内任何一个员工都可以被随时替代，而不影响整体工作进度。Agent 的时代，妥妥一个暗黑老板思维呀。"**

---

## Token 消耗的改善

| 方式 | Token 消耗 |
|------|-----------|
| 单个 Agent 串行 | 全程在一个上下文里跑，容易爆 |
| Agent Team 并行 | 每个 SubAgent 独立上下文，主 Agent 只汇总，消耗更合理 |

---

## 终极总结

> **从 LLM → Agent → Agent Team 的进化逻辑：智力遇到瓶颈就从工程上突破。单个 Agent 遇到瓶颈，就来多个并行。就像芯片上建操作系统一样，我们现在是在大模型这个底座上不断封装工具，终有一天会发现整个世界的交互逻辑都变了模样。**
